📐 Table Calculations

Running totals, percent of total, rank, moving averages, year-over-year growth. Table calculations work on the numbers already in your view rather than the underlying rows — and understanding that one distinction is what separates people who fight Tableau from people who fly in it.

📖 Terms on this page

Defined before they're used. Every term on the site is also in the glossary.

Table calculation
A calculation that runs on the numbers already in your view, not on the source data.
Example: Running total, rank, percent of total, year-over-year growth.
📍 Right-click a measure pill → Quick Table Calculation
Compute Using / addressing
The direction the calculation travels through your table.
Example: Percent of Total computed across makes each row total 100%; down makes each column total 100%.
Partition
The group the calculation restarts within.
Example: A running total that resets each January is partitioned by Year.
Window
The set of cells the calculation can see, set by Compute Using.
Example: WINDOW_AVG averages across that set.
The △ marker
A small triangle Tableau adds to a pill carrying a table calculation.
Example: If a number looks wrong, check for this first — it changes what the pill means.
LOOKUP
A function fetching a value from another position in the table.
Example: LOOKUP(SUM([Sales]), -1) is "the previous period's value".

Why this matters immediately

A table calculation cannot see rows that aren't in your view. Filter out the South region and a "percent of total" calculation now totals only Central, East and West — the percentages change. That's not a bug; it's the definition. It's also the single most common source of confusion, and the reason LOD expressions exist.

🧭 Compute Using — the part everyone gets wrong

A table calculation needs to know which direction to move through the table. Get this wrong and you'll get a plausible-looking number that answers a different question.

Right-click the pill → Compute Using, or Edit Table Calculation for full control.

Table (across)

Moves left → right along each row, restarting on every new row. Use for "% of this row's total".

Table (down)

Moves top → bottom down each column, restarting on every new column.

Table (across then down)

Treats the entire table as one continuous sequence. Use for "% of the grand total".

Pane (down)

Restarts at each pane boundary — e.g. per year rather than across the whole table.

✓ Specific Dimensions

Tick exactly which dimensions the calculation walks along. Explicit, survives rearranging the view, and the option you should reach for once you understand it.

⚠️ Cell

Each cell on its own. Almost never useful — if you land here by accident, your numbers will all be 100% or 1.

What "direction" actually means

The calculation walks through the cells of your table in a path. Compute Using sets that path — and the arrows below are literally it.

Same table, same calculation, two completely different questions answered. Nothing on the chart tells the reader which one you picked — so pick deliberately.

How to check you got it right

Ask what should add up to 100% — rows, columns, or the whole table — then verify that it actually does. If nothing sums to 100%, your Compute Using is wrong. This takes five seconds and catches almost every mistake.

Running Total
Adds each period to everything before it. Always climbs. Answers “how much so far?”
Difference
Each period minus the one before. Turns a level into a change, so ups and downs become obvious.
Percent of Total
Each value as a share. Check they add to 100% — if not, your direction is wrong.
Moving Average
Smooths the noise so the underlying direction shows. Keep the raw line visible behind it.

LOOKUP — reaching to another row

Every “compared to last period” calculation is really a LOOKUP with a negative offset.

Year-over-year growth in four clicks

  1. Build the base view

    YEAR(Order Date) on Columns, SUM(Sales) on Rows.

  2. Add the calculation

    Right-click SUM(Sales) → Quick Table Calculation → Percent Difference.

  3. Check the direction

    Compute Using should be Table (across) — along the years. 2014 will be blank, because there's no prior year to compare to. That blank is correct.

  4. Format it

    Right-click the axis → Format → Percentage, 1 decimal. You now have a growth-rate view that updates with any filter.

Moving averages are underrated

Daily or monthly sales are noisy enough that trends are hard to see. A 3-month moving average on top of the raw line shows the underlying direction while keeping the actual values visible. Add both to the view and make the moving average a thicker, darker line.

✍️ Writing them by hand

Quick Table Calcs cover most cases. When you need more control, write the function directly.

// Running total of sales
RUNNING_SUM(SUM([Sales]))

// Percent of total
SUM([Sales]) / TOTAL(SUM([Sales]))

// Rank — descending by default
RANK(SUM([Sales]))
RANK_DENSE(SUM([Sales]))      // ties share a rank, no gaps
RANK_UNIQUE(SUM([Sales]))     // ties broken arbitrarily

// Difference from the previous period
ZN(SUM([Sales])) - LOOKUP(ZN(SUM([Sales])), -1)

// Percent difference (year-over-year growth)
(SUM([Sales]) - LOOKUP(SUM([Sales]), -1))
    / LOOKUP(SUM([Sales]), -1)

// 3-period moving average (2 back, 0 forward)
WINDOW_AVG(SUM([Sales]), -2, 0)

// Share of the window's maximum — useful for index charts
SUM([Sales]) / WINDOW_MAX(SUM([Sales]))
Table calcs always wrap an aggregate

RUNNING_SUM([Sales]) is invalid — it must be RUNNING_SUM(SUM([Sales])). The inner aggregate produces the view's numbers; the table calc then walks over them. If Tableau complains about mixing aggregate and non-aggregate arguments, this is nearly always why.

LOOKUP — the time-travel function

LOOKUP(expression, offset) fetches a value from another position in the table. It's the engine behind every "compared to last period" calculation.

ExpressionReturns
LOOKUP(SUM([Sales]), -1)The previous period's value
LOOKUP(SUM([Sales]), 1)The next period's value
LOOKUP(SUM([Sales]), -12)Twelve periods back — same month last year
LOOKUP(SUM([Sales]), FIRST())The first value in the window — the base for an index
LOOKUP(SUM([Sales]), LAST())The last value in the window

Offsets that fall outside the window return Null, which is why the first period of a year-over-year view is always blank. Wrap in ZN() only if you genuinely want zero — a blank is usually the more honest answer.

🪟 WINDOW functions and partitioning

Every table calculation operates over a window — the set of cells it can see. Compute Using defines that window, and the WINDOW_* family lets you aggregate across it.

WINDOW_SUM(SUM([Sales]))        // total across the window
WINDOW_AVG(SUM([Sales]))        // average across the window
WINDOW_MAX(SUM([Sales]))        // biggest in the window
WINDOW_MIN(SUM([Sales]))
WINDOW_MEDIAN(SUM([Sales]))
WINDOW_STDEV(SUM([Sales]))

// Optional start/end offsets narrow the window
WINDOW_AVG(SUM([Sales]), -2, 0)   // this period + 2 before
WINDOW_SUM(SUM([Sales]), FIRST(), 0)  // same as RUNNING_SUM

// Positional helpers
INDEX()    // position in the window: 1, 2, 3…
SIZE()     // how many rows are in the window
FIRST()    // offset to the first row (0 when you're on it)
LAST()     // offset to the last row
A genuinely useful trick: INDEX() as a Top N filter

Put INDEX() <= 10 on the Filters shelf with the view sorted descending. Unlike a Top N filter, this one is applied after other filters — so "top 10 within whatever the user selected" works without touching context filters. It's the standard fix for dashboards where the Top N should respond to a filter card.

🔧 When the numbers look wrong

Everything shows 100% or 1

Compute Using is set to Cell. Each cell is its own window, so it's 100% of itself. Change the direction.

Percentages don't sum to 100%

Wrong direction. Decide what should total 100% — a row, a column, the table — and set Compute Using to match.

The first period is blank

Correct behaviour for any "vs previous" calc. There's nothing before the first row.

Numbers change when I filter

Also correct — table calcs only see the view. If you need percentages against the unfiltered total, you need an LOD expression instead.

Rearranging pills broke it

"Table (across)" is defined by layout, so moving pills changes it. Switch to Specific Dimensions to make the calc independent of arrangement.

Totals look wrong

Grand totals compute their own table calc rather than summing the displayed values. Analysis → Totals → Total All Using lets you control this.

The debugging habit that works

Duplicate the sheet and switch it to a plain text table (Show Me → Text Table), with both the raw measure and the table calc side by side. Seeing the input and the output in the same grid makes the direction obvious immediately — far faster than staring at a line chart and guessing.

🧠 Key intuitions

  • Table calcs run on the view, not the data — that's the whole concept
  • They can't see filtered-out rows — by design
  • Compute Using is half the calculation — the function is the other half
  • The △ on a pill means a table calc is applied — always check for it
  • Always wrap an aggregate — SUM([Sales]), never bare [Sales]
  • Ask what should total 100% — then verify it does
  • Specific Dimensions beats Table (across) — it survives layout changes
  • LOOKUP(…, -1) is "the previous one" — the basis of all period comparisons
  • Need it independent of filters? That's an LOD, not a table calc

🏆 Challenge: a Pareto chart

Show that a small share of sub-categories drives most of the sales. Sort sub-categories by Sales descending, add a running total of Sales as a percent of the overall total, and plot it as a line over the bars on a dual axis. Add a reference line at 80%. How many sub-categories does it take to reach it? That single chart is often the most actionable thing in a retail dataset — and it's two table calculations stacked on each other.
