🧮 Measure Names & Measure Values

Two fields Tableau generates for you that most people never really learn — and then wonder why they can't build a simple table with six metric columns in it. This is the mechanism behind any view showing more than one measure at once: scorecards, KPI rows, side-by-side bars, and crosstabs grouped by whatever you like.

Why this trips everyone up

Tableau's normal rule is one pill, one thing. Measure Names and Measure Values deliberately break it: they let a single shelf position carry an unlimited number of measures. Until you know they exist, multi-metric views feel impossible — and once you do, they become routine.

🔍 What they actually are

Scroll to the very bottom of the Data pane. Below a dividing line, Tableau puts two fields that aren't in your data — it generates them.

The line above them is Tableau's way of saying "these are generated, not from your data". Latitude and Longitude show up here too on geographic sources.

Measure Names — a dimension

Its members are the names of your measures, as text: "Sales", "Profit", "Quantity"… Being a dimension, it slices — put it on Columns and you get one column per measure.

Measure Values — a measure

Every measure's numbers stacked into one column. Alone it's meaningless — summing dollars, counts and percentages together is nonsense. It only makes sense sliced by Measure Names, which separates them again.

They're a matched pair. Using one without the other is nearly always a mistake.

Every measure in the source is automatically a member of Measure Names — including calculated fields you add later. Build a margin calculation and it appears in the list on its own.

🔄 The one idea: they unpivot your measures

This is what makes everything else obvious. Measure Names and Measure Values perform an unpivot — the same reshape as Power Query's Unpivot, done on the fly and invisibly. Follow the colours to trace where each number goes.

Two rows became six. "Which measure" is now a value in a column — so it's a dimension.

Everything follows from this

Once "which measure" is a dimension, you can do to it anything you'd do to any other dimension:

  • Put it on Columns → one column per measure (a table)
  • Put it on Rows → one row per measure (a vertical KPI list)
  • Put it on Color → each measure gets its own colour and a legend
  • Put it on Filters → choose which measures appear
  • Combine it with another dimension → grouped bars

That's the whole feature. Every pattern below is one of those five.

🏗️ Build it, one action at a time

Follow along. Each frame is the view after exactly one more action.

The recipe, in one line

Dimensions on Rows = your grouping · Measure Names on Columns = your metric columns · Measure Values on Text = the numbers. Swap the dimensions freely — the metric block stays put. That's the answer to "group by something and put a bunch of columns against it".

🎨 Colour-coding the cells

A table of numbers is precise but flat. Colour gives it the at-a-glance quality of Excel's conditional formatting — and there's one trap that catches everybody on the first attempt.

The fix: right-click the Measure Values pill on Color → Use Separate Legends.

The better approach — colour by a rule

Usually preferable, because you decide what "good" means rather than letting the data's range decide. Write a calculation and put that on Color instead:

// A three-state traffic light on margin
IF     SUM([Profit])/SUM([Sales]) < 0    THEN "Losing money"
ELSEIF SUM([Profit])/SUM([Sales]) < 0.10 THEN "Thin margin"
ELSE                                      "Healthy"
END

Why you can't add Margin to that chart

All the bars share one axis. Sales, Profit and Cost together are fine — all dollars. Add a margin of 0.15 to an axis scaled for $725,000 and here is the literal result:

Not a bug. A shared axis means a shared scale, and 0.149 really is invisible next to 725,458. When units differ, use a dual axis, separate rows, or go back to a text table.

⚖️ Shared vs dual vs separate axes

Three ways to get multiple measures into one chart. They look similar and behave very differently.

Shared axis
Measure Names + Values. Any number of measures, one scale, one Marks card. Units must match.
Dual axis
Exactly two measures, two overlaid axes, two Marks cards — so different mark types. Units may differ.
Separate axes
Several pills left alone. One pane per measure, each with its own scale. Units may differ freely.
Deciding in one question

Are all the measures in the same units?
Yes → shared axis. · No, and there are exactly two → dual axis, both axes labelled. · No, and there are more than two → separate axes, or a text table.

💲 Per-measure number formatting

A scorecard mixing dollars, percentages and counts needs each column formatted differently. Two places to set it, and the difference matters.

✓ On the field — do this

Right-click the field in the Data pane:

Default Properties → Number Format

Applies wherever that field is ever used, in this and every future sheet. Set it once when you build the calculation.

⚠️ On the pill — local only

Right-click a pill in the Measure Values card → Format.

This view only. Fine for a one-off override; you'll redo it on every other sheet.

Text tables handle mixed formats; shared axes can't

In a text table every cell renders in its own measure's format, so dollars, percentages and counts sit happily side by side. On a shared axis there's only one axis and therefore one format — another reason the text table is the right home for a mixed scorecard.

🔠 Sorting a multi-measure table

Clicking a column header — the thing everyone tries first — doesn't reliably work here, because those "columns" are members of a dimension rather than fields. Sort explicitly instead.

  1. Right-click the dimension you want to reorder

    The pill on Rows — Region, or Sub-Category. Not the measure.

  2. Sort → By Field

    Set direction to Descending, then pick the specific measure — Profit — from the Field Name dropdown, with the right aggregation.

  3. Check the sort icon appeared on the pill

    Unlike a toolbar quick-sort, this definition survives filtering and being placed on a dashboard.

The metric columns are not sorted — they're ordered

Left-to-right column order comes purely from the pill order in the Measure Values card. Drag the pills to change it. Tableau will not sort them for you, and there's no alphabetical option.

➕ Totals and subtotals

Analysis → Totals → Show Grand Totals adds a total row; Show Subtotals adds one per group when dimensions are nested. Each metric column totals its own measure.

⚠️ Ratio columns need checking

A Margin total should be total profit ÷ total sales — not the sum of the regional margins, and not their average. Whether Tableau gets it right depends on how you wrote it:

  • SUM([Profit])/SUM([Sales]) — an aggregate expression, recomputed at the total level. Correct.
  • AVG([Profit]/[Sales]) — a row-level ratio, averaged. Wrong at every level.

If a total still looks off, right-click it → Total Using. Leave it on Automatic for properly-written aggregate ratios — that's what makes them recompute rather than add up.

🎚️ Letting viewers choose the metrics

Because Measure Names is an ordinary dimension, it can go on Filters — and a filter can be shown as a control.

  1. Put Measure Names on Filters

    Tick the measures that should be available. This also prunes the columns down from "everything in the source".

  2. Show Filter

    Right-click the pill → Show Filter, then set it to Multiple Values (list).

  3. Watch the table reshape

    Columns appear and disappear as they tick. One sheet serves the person who wants two metrics and the one who wants seven.

Related, but different: the measure swapper

Measure Names controls which of several metrics appear at once. A parameter plus a CASE calculation swaps which single metric a chart shows — see Calculated Fields. Measure Names for tables; a parameter for single-metric charts.

🕳️ Traps

Every measure appears at once

Tableau includes all of them, including Row ID and Postal Code if those were left as measures. Prune the card, and fix the offending fields in the Data pane.

One shared colour scale flattens everything

Colouring by Measure Values puts every measure on one range. Right-click the Color pill → Use Separate Legends.

Small-unit bars vanish

A shared axis scaled for $725K makes a 0.15 margin bar sub-pixel. Don't mix units on one axis.

Table calculations misbehave

Measure Names is a dimension in the view, so it's part of the addressing. Set Compute Using to Specific Dimensions and be explicit.

You can't reference them in a calculation

They're generated at view time, so [Measure Names] isn't available in the editor. Logic conditional on "which measure" needs a parameter.

Grand totals on ratios

Correct only when the ratio is an aggregate expression. Always check the total row against a number you trust.

🧠 Key intuitions

  • They unpivot your measures — "which measure" becomes just another dimension
  • Names is a dimension, Values is a measure — they only work as a pair
  • Dimensions on Rows, Measure Names on Columns, Measure Values on Text is the scorecard recipe
  • The Measure Values card is the control panel — include, exclude, reorder, re-aggregate
  • Column order = pill order on that card, and nothing else
  • Add the same measure twice with different aggregations for sum-and-average columns
  • Set number formats on the field, not the pill, so they follow it everywhere
  • Use Separate Legends whenever you colour by Measure Values
  • Shared axis only when units match — otherwise dual axis or a table
  • Sort by right-clicking the dimension → Sort → By Field → pick the measure
