🧮 Measure Names & Measure Values

Two fields Tableau generates for you that most people never really learn — and then wonder why they can't build a simple table with six metric columns in it. This is the mechanism for any view that shows more than one measure at once: scorecards, KPI rows, side-by-side bars, and crosstabs grouped by whatever you like.

🧩 The problem

You want a table: Region down the side, and Sales, Profit, Margin, Orders and Average Order Value across the top. An ordinary business table. In a spreadsheet it takes thirty seconds.

In Tableau, you drag Region to Rows, then Sales to Columns, then Profit to Columns — and instead of two columns you get two separate charts with two separate axes, stacked side by side. Add three more measures and it gets worse.

Five measure pills on Columns

Five independent axes, five panes, each scaled to its own range. Not a table — five charts wearing a trenchcoat.

And no way to sort by one of them, or colour cells by value.

→
Measure Names on Columns

One column per measure, one row per region, every cell formatted in its own units. Sortable, colourable, totalled.

Two pills, any number of measures.

Why this trips everyone up

Tableau's normal rule is one pill, one thing. Measure Names and Measure Values break that rule deliberately: they let a single shelf position carry an arbitrary number of measures. Until you know they exist, multi-metric views feel impossible — and once you do, they become routine.

🔍 What they actually are

Scroll to the very bottom of the Data pane, below a dividing line. Tableau puts two fields there that aren't in your data — it generates them:

Measure Names — a dimension

A dimension whose members are the names of your measures, as text.

Its members
Sales
Profit
Quantity
Discount
Profit Margin (your calc)

Being a dimension, it slices — put it on Columns and you get one column per measure.

Measure Values — a measure

A single measure containing every measure's numbers, stacked into one column.

On its own it's meaningless — summing dollars, counts and percentages together gives nonsense. It only makes sense sliced by Measure Names, which separates the values back out.

They're a matched pair. Using one without the other is almost always a mistake.

Every measure in your data source is automatically a member of Measure Names — including calculated fields you add later. Create a new margin calculation and it appears in the list without you doing anything.

🔄 The mental model: they unpivot your measures

This is the single idea that makes everything else obvious. Measure Names and Measure Values perform an unpivot — the same reshape as Power Query's Unpivot, done on the fly, invisibly.

Everything follows from this

Once "which measure" is a dimension, you can do to it anything you'd do to any other dimension: put it on Columns to make table columns, on Rows to stack them, on Color to give each measure a colour, on Filters to choose which appear, and sort by it. That's the whole feature.

➕ Three ways to get them into a view

  1. Drop a second measure onto an existing axis

    Build Sales by Region, then drag Profit and hover over the Sales axis itself until a small green double-bar icon appears. Drop it there. Tableau silently swaps in Measure Names and Measure Values for you, and puts Measure Names on Color.

    This is the accidental route — it's how most people meet these fields without realising what happened.

  2. Double-click Measure Values in the Data pane

    Tableau drops it into the view and brings Measure Names along, adding it to the Filters shelf so you can pick which measures are included. Fastest route to a scorecard.

  3. Place both by hand — the one to learn

    Drag Measure Names to Columns and Measure Values to Text. You now control exactly where each one sits, which is what you want for anything deliberate.

Know which one happened to you

If a chart suddenly grows a colour legend listing your measure names, route 1 fired. That's fine — but check the Measure Values card immediately, because Tableau will have included every measure in the data source, including ones you don't want.

📋 Building the scorecard, step by step

The view you actually wanted: grouped by whatever you like, with a set of metric columns beside it. Here's the whole thing.

  1. Build the metrics you need first

    Some columns aren't raw fields. Create these as calculated fields before you start, so they show up as members of Measure Names:

    // Profit Margin — aggregate, not row-level
SUM([Profit]) / SUM([Sales])

// Average Order Value — COUNTD, not COUNT
SUM([Sales]) / COUNTD([Order ID])

// Orders — a distinct count of order IDs
COUNTD([Order ID])
  2. Put your grouping dimensions on Rows

    Drag Region to Rows. Then drag Category to Rows after it — you now have nested grouping, Region containing Category. Add a third if you want; the metric columns don't change.

  3. Put Measure Names on Columns

    Drag Measure Names to Columns. You'll get a column for every measure in the source — far too many. That's expected.

  4. Put Measure Values on Text

    Drag Measure Values onto Text in the Marks card. Set the mark type to Text if it isn't already. The numbers appear.

  5. Prune and order the columns

    On the Measure Values card, drag off everything you don't want, then reorder what's left. Sales, Profit, Margin, Orders, AOV — in that order, because it reads as a narrowing story.

  6. Format each measure

    Right-click each field in the Data pane → Default Properties → Number Format. Set Sales and Profit to currency with no decimals, Margin to percentage with one, Orders to a plain number with a thousands separator. Because this is set on the field itself, every future view inherits it.

  7. Clean up the frame

    Right-click the "Measure Names" field label above the columns → Hide Field Labels for Columns. Do the same for Rows if the dimension names are obvious. The table is now just headers and numbers.

This is the pattern to internalise

Dimensions on Rows = your grouping. Measure Names on Columns = your metric columns. Measure Values on Text = the numbers. Swap the dimensions freely; the metric block stays put. That's the answer to "group by something and put a bunch of columns against it".

🎨 Colour-coding the cells — Tableau's conditional formatting

A table of numbers is precise but flat. Adding colour gives it the at-a-glance quality of Excel's conditional formatting, and the mechanism is the same idea: a rule, not a paint stroke.

Option A — colour by the value itself

Drag Measure Values onto Color as well as Text, and change the mark type to Square. Every cell is now shaded by its own number.

⚠️ And it will look wrong — here's why

All measures share one colour scale. Sales runs to 700,000 and Margin runs to 0.15, so on a single scale every margin cell is at the bottom of the range and every one comes out the same colour. The colour is technically correct and completely useless.

The fix: right-click the Measure Values pill on Color → Use Separate Legends. Tableau then gives each measure its own colour scale and its own legend, and the shading becomes meaningful per column.

Option B — colour by a rule you write

Usually better, because you decide what "good" means rather than letting the data's range decide. Write a calculation and put that on Color:

// Simple: red when we're losing money
SUM([Profit]) < 0

// Banded: a three-state traffic light on margin
IF    SUM([Profit])/SUM([Sales]) < 0    THEN "Losing money"
ELSEIF SUM([Profit])/SUM([Sales]) < 0.10 THEN "Thin margin"
ELSE                                     "Healthy"
END

// Against a target the viewer controls
IF SUM([Sales]) >= [Target Parameter] THEN "On track"
ELSE "Behind" END

A discrete calculation like the middle one gives you a clean three-colour legend beside the chart, with labels a reader understands — rather than a continuous ramp they have to interpret.

Controlling that legend

Show or hide it from the sheet's ▾ → Legends menu. Rename it by right-clicking → Edit Title — "Margin band" reads better than "Margin Status". Option C — bars inside the table

The closest equivalent to Excel's data bars. Build the text table, then add a second measure on a dual axis with a Bar mark behind the text — or more simply, make one column a bar chart by putting Measure Values on Size or building a separate narrow sheet beside the table on a dashboard. For a single metric, the standard approach is a highlight table (Square marks + Color), which is what Option A produces once you've fixed the separate-legends problem.

📊 Side-by-side bars

Same two fields, different shelves. Instead of a table you get grouped bars — one cluster per category, one bar per measure.

Measure Names appears twice — on Columns to create the groups, and on Color so each measure is identifiable.

This only works when the units match

All the bars share one axis. Sales, Profit and Cost together are fine — all dollars, comparable heights. Put Sales and Margin on the same axis and the margin bars are invisible slivers at zero. When units differ, use a dual axis, separate rows, or go back to a text table.

Deciding in one question

Are all the measures in the same units? Yes → shared axis. No, and there are exactly two → dual axis with both axes labelled. No, and there are more than two → separate axes, or a text table. That question resolves it almost every time.

💲 Per-measure number formatting

A scorecard mixing dollars, percentages and counts needs each column formatted differently. There are two places to set that, and the difference matters.

✓ On the field — do this

Right-click the field in the Data pane:

Default Properties → Number Format

Applies everywhere that field is ever used, in this and every future sheet. Set it once when you build the calculation and never think about it again.

⚠️ On the pill — local only

Right-click a pill in the Measure Values card → Format.

Applies to this view only. Useful for a one-off override — showing Sales in thousands on a cramped dashboard tile — but it won't follow the field elsewhere, and you'll redo it on the next sheet.

Text tables handle mixed formats; shared axes can't

In a text table, each cell renders in its own measure's format — dollars, percentages and counts sit happily side by side. On a shared axis there is only one axis, so there's only one format, and mixing units was already a mistake for scale reasons. Another reason the text table is the right home for a mixed scorecard.

Custom format strings — $#,##0,"K", ▲0.0%;▼0.0%;— — are covered in detail in Formatting & Custom Charts.

🔠 Sorting a multi-measure table

Clicking a column header to sort — the thing everyone tries first — doesn't reliably work here, because the "columns" are members of a dimension rather than fields. Do it explicitly instead.

  1. Right-click the dimension you want to reorder

    The pill on Rows — Region, or Sub-Category. Not the measure.

  2. Choose Sort

    Set Sort By to Field, direction to Descending, and then pick the specific measure — Profit, say — from the Field Name dropdown, with the right aggregation.

  3. Check it held

    A small sort icon appears on the pill. Unlike a toolbar quick-sort, this definition survives filtering and being placed on a dashboard.

Sorting the measure columns themselves

The left-to-right order of the metric columns is not a sort — it's the order of the pills in the Measure Values card. To rearrange them, drag the pills. Tableau will not sort them for you, and there's no alphabetical option.

➕ Totals and subtotals

Analysis → Totals → Show Grand Totals adds a total row; Show Subtotals adds one per group when you have nested dimensions. Each metric column totals its own measure, which is what you want.

⚠️ Ratio columns need checking

A Margin column's grand total should be total profit ÷ total sales — not the sum of the four regional margins, and not their average. Whether Tableau gets this right depends on how you wrote the calculation:

  • SUM([Profit])/SUM([Sales]) — an aggregate expression, recomputed at the total level. Correct.
  • AVG([Profit]/[Sales]) — a row-level ratio, averaged. Wrong at every level, including the total.

If a total still looks off, right-click it → Total Using and set it explicitly. Leave it on Automatic for properly-written aggregate ratios — that's what makes them recompute rather than add up.

Grand totals default to the bottom and right. Move them with Analysis → Totals → Row Totals to Left — worth doing when the total is the number people came for.

🎚️ Letting viewers choose the metrics

Because Measure Names is an ordinary dimension, it can go on the Filters shelf — and a filter can be shown as a control.

  1. Put Measure Names on Filters

    Tick the measures that should be available. This also prunes the columns down from "everything in the data source".

  2. Show the filter

    Right-click the pill → Show Filter. Set it to Multiple Values (list) so viewers can tick metrics on and off.

  3. Watch the table reshape

    Columns appear and disappear as they choose. One sheet now serves the person who wants two metrics and the one who wants seven.

Related, but different: the measure swapper

Measure Names controls which of several metrics appear. A parameter plus a CASE calculation swaps which single metric a chart displays — see Calculated Fields. Use Measure Names for tables, a parameter for single-metric charts where you want one axis that changes meaning.

🕳️ Traps

Every measure appears at once

Tableau includes all of them by default, including Row ID and Postal Code if those were left as measures. Prune the Measure Values card immediately, and fix the offending fields in the Data pane.

One shared colour scale flattens everything

Colouring by Measure Values puts all measures on one range. Right-click the Color pill → Use Separate Legends.

Bars for small-unit measures vanish

A shared axis scaled for $700K makes a 0.15 margin bar a single pixel. Don't mix units on one axis.

Table calculations behave unexpectedly

Measure Names is a dimension in the view, so it's part of the addressing. Set Compute Using to Specific Dimensions and be explicit about whether Measure Names is included.

You can't reference them in a calculation

They're generated at view time, so [Measure Names] isn't available in the calculation editor. Anything conditional on "which measure" has to be handled with a parameter instead.

Grand totals on ratios

Correct only if the ratio is written as an aggregate expression. Always check the total row against a number you trust.

🧠 Key intuitions

  • They unpivot your measures — "which measure" becomes just another dimension
  • Names is a dimension, Values is a measure — they only work as a pair
  • Dimensions on Rows, Measure Names on Columns, Measure Values on Text is the scorecard recipe
  • The Measure Values card is the control panel — include, exclude, reorder, re-aggregate
  • Column order = pill order on that card, and nothing else
  • Add the same measure twice with different aggregations for sum-and-average columns
  • Set number formats on the field, not the pill, so they follow it everywhere
  • Use Separate Legends whenever you colour by Measure Values
  • Shared axis only when units match — otherwise dual axis or a table
  • Sort by right-clicking the dimension → Sort → By Field → pick the measure

🏆 Challenge: a full regional scorecard

Build a single sheet with Region and Sub-Category nested on Rows and seven metric columns: Sales, Profit, Margin, Orders, Average Order Value, Average Discount, and Sales as a percent of total (a table calculation added on the Measure Values card). Format every column in its own units. Colour the Margin column only — using a calculation, so the other columns stay plain. Add subtotals per region and a grand total, and confirm the Margin figures recompute rather than adding up.

Then hide the field labels, strip the gridlines, and put a Measure Names filter on the sheet so a viewer can hide columns they don't care about. That's a production-quality table, and it's two special fields plus a lot of small deliberate choices.
