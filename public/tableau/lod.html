🔬 LOD Expressions

Level of Detail expressions let a calculation aggregate at a different granularity than the view it sits in. They solve the problems that make people give up on Tableau — customer-level metrics inside a regional chart, percentages that ignore filters, cohort analysis. They're also the most-asked Tableau interview topic.

📖 Terms on this page

Defined before they're used. Every term on the site is also in the glossary.

Level of Detail (LOD)
The granularity a number is calculated at — how finely the data is cut before summing.
Example: The view might be per region while you need a figure computed per customer.
LOD expression
A calculation, written in curly braces, that aggregates at a different granularity than the view.
Example: {FIXED [Customer Name] : SUM([Sales])} — each customer's lifetime spend.
FIXED
Computes at exactly the dimensions you name, ignoring what's in the view.
Example: Stays per-customer even inside a regional chart. The predictable one.
INCLUDE
Starts from the view's granularity and goes finer.
Example: Average order size, computed per order then averaged at whatever level you're showing.
EXCLUDE
Starts from the view's granularity and goes coarser.
Example: A regional total displayed inside a chart broken down by category.
Cohort
A group defined by when they first appeared, tracked over time.
Example: Customers grouped by the year of their first order, to see whether they keep buying.

🧩 The problem LODs solve

Tableau aggregates to whatever dimensions are in your view. Put Region on Rows and you get four numbers. That's usually what you want — until it isn't.

A question you cannot answer without LODs

"What is the average customer's total spend, per region?"

You need to sum sales per customer first, then average those totals per region. Two levels of aggregation in one number. AVG([Sales]) gives you the average line item, which is a completely different — and much smaller — figure.

The average line item. Not what was asked.

→

Total spend per customer, then averaged.

// Step 1 — a calculated field that sums per customer,
//          regardless of what's in the view
// Name: Customer Total
{ FIXED [Customer Name] : SUM([Sales]) }

// Step 2 — drop it on the view and set the aggregation to AVG
AVG([Customer Total])

📝 The syntax

{ KEYWORD [dimensions] : AGGREGATE(expression) }
  • { } Curly braces mark it as an LOD — always required
  • KEYWORD FIXED, INCLUDE or EXCLUDE
  • [dimensions] The granularity to aggregate at. Comma-separated; can be empty
  • : Separates the dimensions from the calculation
  • AGGREGATE(…) SUM, AVG, MIN, MAX, COUNTD — required, not optional
{ FIXED [Region] : SUM([Sales]) }
{ FIXED [Region], [Category] : SUM([Sales]) }   // two dimensions
{ FIXED : SUM([Sales]) }                     // no dimensions = grand total
{ INCLUDE [Customer Name] : SUM([Sales]) }
{ EXCLUDE [Category] : SUM([Sales]) }
Read it out loud

{ FIXED [Region] : SUM([Sales]) } reads as "the sum of sales, fixed at the region level". Whatever else is in the view, that number is computed per region and nothing else.

The three keywords on one scale

Every LOD does the same thing: it moves the calculation to a different point on the granularity scale. The only difference is how it picks the point.

  1. I
    INCLUDE — starts from wherever the view is and moves finer. Relative: change the view and it moves too.
  2. E
    EXCLUDE — starts from the view and moves coarser. Also relative.
  3. F
    FIXED — ignores the view entirely and computes at the level you named. Absolute, which is why it's the predictable one.
  4. ?
    Nothing here changes what is calculated — only the granularity it's calculated at.
Why FIXED is the one you'll reach for

INCLUDE and EXCLUDE are defined relative to the view, so the same expression means different things on different sheets — powerful, but harder to reason about. FIXED names an absolute level, so it computes the same number no matter what you drag around. Start with FIXED; use the other two when you specifically want the number to adapt.

📌 FIXED — ignore the view entirely

The one you'll use 90% of the time. FIXED computes at exactly the dimensions you name and ignores every other dimension in the view.

Category is in the view, but the FIXED expression ignores it — every Central row shows the whole Central total. That repetition is exactly what makes "% of region" possible.

What that unlocks

// Each category's share of ITS OWN region
SUM([Sales]) / MIN({ FIXED [Region] : SUM([Sales]) })

// Share of the grand total — no dimensions at all
SUM([Sales]) / MIN({ FIXED : SUM([Sales]) })

// A customer's lifetime value, usable at any level
{ FIXED [Customer Name] : SUM([Sales]) }

// The date each customer first ordered — the basis of cohort analysis
{ FIXED [Customer Name] : MIN([Order Date]) }

// How many orders has each customer placed?
{ FIXED [Customer Name] : COUNTD([Order ID]) }
Why the MIN() wrapper?

An LOD returns one value per group, but Tableau still needs an aggregate to combine it into the view. MIN, MAX and AVG all return the same answer here because every row in the group holds the identical value — MIN is just the convention. Omit it and you'll get "cannot mix aggregate and non-aggregate arguments".

➕ INCLUDE — go finer than the view

Adds dimensions to whatever is already in the view, computes there, then aggregates back up. Use it when you need a more detailed level than you're displaying.

// Average order size, shown at whatever level the view is at
AVG({ INCLUDE [Order ID] : SUM([Sales]) })

// Reads as: sum sales per order (plus whatever's in the view),
// then average those order totals.
INCLUDE vs FIXED — the practical difference

INCLUDE adapts to the view: drag Region in and it computes per order within each region. FIXED ignores the view entirely. If you want a number that responds to how you slice the dashboard, use INCLUDE; if you want a constant reference value, use FIXED.

➖ EXCLUDE — go coarser than the view

Removes dimensions from the view's granularity. The classic use is showing a total alongside its own breakdown.

// Regional total, displayed in a view broken down by Category
{ EXCLUDE [Category] : SUM([Sales]) }

// Each category's share of its region — an alternative to the FIXED version
SUM([Sales]) / MIN({ EXCLUDE [Category] : SUM([Sales]) })

// How far is this month above or below the yearly average?
SUM([Sales]) - AVG({ EXCLUDE [Month of Order Date] : SUM([Sales]) })

EXCLUDE is relative — it strips named dimensions from whatever's in the view. That makes it flexible but harder to reason about, which is why most people reach for FIXED. Both are correct; FIXED is easier to debug.

🪜 Order of operations — where FIXED surprises people

Tableau evaluates things in a fixed sequence. FIXED is computed before dimension filters, which is either exactly what you want or a nasty surprise.

  1. Extract filters
    Applied when the data was extracted
  2. Data source filters
    Set on the connection
  3. Context filters
    Right-click a filter → Add to Context. These DO affect FIXED.
  4. FIXED LOD expressions
    Computed here — before ordinary filters
  5. Dimension filters
    Your normal filter cards. FIXED ignores these.
  6. INCLUDE / EXCLUDE LODs
    Computed after dimension filters, so they respect them
  7. Measure filters
    Filters on aggregated values
  8. Table calculations
    Run last, on the finished view
⚠️ The behaviour that confuses everyone

Filter your dashboard to the West region. A {FIXED : SUM([Sales])} grand total still shows $2,297,201 — all four regions — because it was computed before your filter ran.

Sometimes that's the goal ("what share of company-wide sales is West?"). When it isn't, right-click the Region filter → Add to Context, and the FIXED expression will then respect it.

✓ Want the LOD to ignore filters

Use FIXED and leave the filters alone. Good for "% of company total" or a constant benchmark line.

✓ Want the LOD to respect filters

Either add the filter to Context, or use INCLUDE / EXCLUDE, which sit after dimension filters in the sequence.

⚖️ LOD or table calculation?

They overlap, and choosing wrongly is the source of a lot of frustration. The deciding question: does the number need to see data that isn't in the view?

The rule of thumb

Anything positional — running total, rank, "compared to the previous row", moving average — is a table calculation. Anything about granularity — per customer, per order, ignoring a filter — is an LOD.

📖 Recipes worth memorising

Customer cohort analysisGroup customers by when they first bought
// Acquisition date — the customer's first ever order
{ FIXED [Customer Name] : MIN([Order Date]) }

// Cohort year — turn that into a grouping dimension
YEAR({ FIXED [Customer Name] : MIN([Order Date]) })

// Is this order the customer's first?
IF [Order Date] = { FIXED [Customer Name] : MIN([Order Date]) }
THEN "New" ELSE "Returning" END

Put Cohort Year on Rows and Order Year on Columns and you have a retention grid — the analysis every subscription business runs, and it's three lines of calculation.

Customer segmentationClassify by total behaviour, not per row
// Bucket customers by lifetime spend
IF { FIXED [Customer Name] : SUM([Sales]) } > 5000 THEN "VIP"
ELSEIF { FIXED [Customer Name] : SUM([Sales]) } > 1000 THEN "Regular"
ELSE "Occasional"
END

// Repeat customers only
{ FIXED [Customer Name] : COUNTD([Order ID]) } > 1

Because it's FIXED, the segment stays stable no matter how you slice the view — a customer doesn't stop being a VIP because you filtered to one region.

Percent of total that ignores filtersThe table-calc problem, solved
// Share of company-wide sales, even when filtered
SUM([Sales]) / MIN({ FIXED : SUM([Sales]) })

// Compare with the table-calc version, which uses only
// what survived the filters:
SUM([Sales]) / TOTAL(SUM([Sales]))

Put both on the same view with a filter active and the difference becomes obvious in one glance. Which is "right" depends entirely on the question you're answering.

Comparison against a benchmarkEvery row measured against a constant
// How does this sub-category compare to the overall average?
SUM([Profit]) - MIN({ FIXED : AVG([Profit]) })

// Above or below its own region's average?
SUM([Sales]) - MIN({ FIXED [Region] : AVG([Sales]) })

// Percent of the best-performing region
SUM([Sales]) / MAX({ FIXED [Region] : SUM([Sales]) })

🧠 Key intuitions

  • An LOD changes the granularity a number is computed at — nothing else
  • FIXED ignores the view; INCLUDE goes finer; EXCLUDE goes coarser
  • Read it as a sentence — "sum of sales, fixed at customer level"
  • FIXED runs before dimension filters — Context filters are how you override that
  • Wrap LODs in MIN/MAX/AVG when combining them with other aggregates
  • Positional → table calc, granularity → LOD
  • {FIXED : …} with no dimensions is a grand total that survives filtering
  • FIXED makes segments stable — a VIP stays a VIP whatever you filter to

🎯 Practice exercises

  1. Create Customer Total = {FIXED [Customer Name] : SUM([Sales])} and show its AVG by Region
  2. Compare that against plain AVG([Sales]) in the same view — explain the gap in one sentence
  3. Build a % of grand total using {FIXED : SUM([Sales])}
  4. Add a Region filter and confirm those percentages don't change — then explain why
  5. Add the Region filter to Context and watch them change
  6. Build each category's share of its own region using {FIXED [Region] : SUM([Sales])}
  7. Create Customer Acquisition Date with MIN([Order Date]) fixed per customer
  8. Turn that into a Cohort Year dimension and build a cohort × order-year grid
  9. Create a New vs Returning flag and chart the split over time
  10. Segment customers into VIP / Regular / Occasional by lifetime spend
  11. Find customers with more than one order using COUNTD inside a FIXED
  12. Show each sub-category's profit as a difference from the overall average
🏆 Challenge: are we growing by acquiring or by keeping?

Build a view splitting each year's revenue into new-customer revenue and returning-customer revenue, using a FIXED acquisition date to classify each order. Then answer the question a board actually cares about: is revenue growth coming from customers we already had, or from ones we bought? Add the count of distinct customers per cohort and you can see retention decay directly. This analysis is genuinely difficult without LODs, and near-trivial with them — which is the best possible argument for learning them.
