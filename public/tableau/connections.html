🔌 Connecting Data

Before you can visualise anything you have to get the data in — and get its shape right. Joins, relationships, unions, live vs extract: these decisions are made once at the start and quietly determine whether your numbers are correct for the rest of the project.

🔗 What Tableau can connect to

📄 Files

Excel, CSV, JSON, PDF, spatial files, statistical files. What you'll use while learning.

🗄️ Databases

SQL Server, PostgreSQL, MySQL, Snowflake, BigQuery, Redshift. What you'll use at work.

☁️ Cloud services

Google Sheets, Salesforce, Google Analytics — plus a generic web-data connector.

Tableau Public limitation: the free version connects to files, Google Sheets and a few web sources — not to databases. That's fine for everything in this course, and the concepts transfer unchanged to Tableau Desktop.

🪢 Relationships — the modern default

Since 2020.2, dragging a second table onto the canvas creates a relationship (a flexible noodle line) rather than a join. This is the default, and usually the right choice.

✓ What a relationship does

  • Keeps tables separate and at their own granularity
  • Tableau writes the right join per visualisation, based on the fields you use
  • No duplicated rows from one-to-many fan-out
  • Aggregates stay correct automatically

When to use it

Almost always. Especially when tables have different granularity — Orders (one row per line item) related to Returns (one row per returned order), or to Targets (one row per region per month).

Why this was a big deal

Before relationships, connecting a monthly Targets table to line-item Orders would duplicate every target across thousands of rows, and SUM(Target) would come out wildly inflated. People lost entire afternoons to it. Relationships solve that by keeping the tables apart until the moment a specific view needs them combined.

⛓️ Joins — when you need row-level control

Double-click a table on the canvas to open the physical layer, where you can build a traditional join. Joins merge tables into one flat table immediately, before any visualisation.

Inner
Only rows matching in both tables. Silently drops unmatched rows — check your row count.
Left
Everything from the left table, matches from the right, Null where none. The safe default.
Right
The mirror image. Rare — most people swap the tables and use a left join instead.
Full outer
Every row from both sides. Useful for reconciliation — finding what exists in one system but not the other.
Left join is almost always the right starting point

It guarantees you keep every row of your main table. If the join key is broken you'll see Nulls — visible, diagnosable evidence. An inner join in the same situation silently deletes rows, and your totals come out low with no indication why.

SUM(Target) = 1,400. The real total is 900. The West target got counted twice because West had two order rows.

Three ways to avoid it

  1. Use a relationship instead of a join

    The modern fix, and the reason relationships exist. Tableau keeps the tables separate and aggregates each at its own granularity. SUM(Target) returns 900.

  2. Or aggregate before joining

    If you must join, roll the finer table up to the coarser table's granularity first so the join is one-to-one.

  3. Or de-duplicate with an LOD

    MIN({FIXED [Region] : MIN([Target])}) pulls one value per region regardless of how many times the row repeats. See LOD Expressions.

Always check the row count after joining

Drop Number of Records (or COUNT(*)) into a view before and after. If it grew, you have fan-out. Superstore's Orders table has 9,994 rows — if it's larger after you connect Returns, something duplicated.

📚 Unions — stacking, not widening

A join adds columns. A union adds rows. Use it when the same data arrives split across several files or tabs — one per month, one per region.

Drag a table below an existing one on the canvas, or right-click → Convert to Union. Use a wildcard union to pick up every sheet or file matching a pattern — new files then join automatically on refresh.

Columns match by name

"Order Date" and "OrderDate" become two separate columns, each half full of Null. Tableau adds a Table Name field showing each row's origin — useful both for debugging and for extracting the month or region encoded in a filename.

⚡ Live vs Extract

A choice you make on the Data Source page, top right. It affects speed, freshness and what features are available.

Live
  • Queries the source every interaction
  • Always current
  • Speed depends entirely on the database
  • Needs a constant connection

Use when data changes constantly and freshness matters more than speed.

Extract (.hyper)
  • A compressed local snapshot
  • Usually much faster
  • Works offline
  • Needs refreshing to update

Use when the source is slow, remote, or you're publishing to Tableau Public — which requires an extract.

Extracts can be filtered and aggregated

You don't have to extract everything. Edit → Filters limits which rows are pulled; Aggregate data for visible dimensions pre-rolls it up. A 50-million-row table becomes a fast few-thousand-row extract when you only ever look at monthly regional totals.

🧹 Cleaning at the source

Fixes made on the Data Source page apply everywhere, forever. Fixes made in a worksheet apply to that worksheet. Prefer the former.

Rename fields

Double-click a column header. Do this for anything cryptic — the names appear on every axis and tooltip you ever build.

Fix data types

Click the icon above a header. Text that should be a date, numbers stored as strings, IDs that shouldn't be summed.

Hide unused fields

Right-click → Hide. Superstore's Row ID and Country (always "United States") add nothing.

Split columns

Right-click → Split, or Custom Split. Turns "CA-2016-152156" into usable parts without a calculated field.

Data source filters

Top-right → Filters → Add. Excludes rows from everything — test records, cancelled orders, irrelevant years.

Set geographic roles

Right-click → Geographic Role. Needed before a field will work on a map.

Data source filters are invisible later

They're powerful and easy to forget. Six months on, someone asks why the totals don't match the source system and the answer is a filter set on this page that nobody remembers. Document them — a text object on the dashboard, or a note in the workbook description.

🧠 Key intuitions

  • Relationships by default — joins only when you need row-level control
  • Joins add columns; unions add rows
  • Left join is the safe join — Nulls are visible, dropped rows aren't
  • Always check the row count after connecting — growth means fan-out
  • Different granularity is the danger — that's where duplication comes from
  • Fix names and types once, at the source
  • Extract for speed, live for freshness — Tableau Public requires an extract
  • Document data source filters — they're invisible from the worksheet

🎯 Practice exercises

The Superstore workbook ships with three tables: Orders, Returns and People.

  1. Open the Data Source page and note the row count of Orders (it should be 9,994)
  2. Relate Returns to Orders on Order ID, then confirm the row count hasn't changed
  3. Build a view of Sales by Region and confirm the total is still $2,297,201
  4. Now join Returns instead of relating it — check the row count again and explain any change
  5. Create a Returned? calculation flagging orders that appear in Returns
  6. Relate the People table and show Sales by regional manager
  7. Rename three fields to something clearer and hide Row ID and Country
  8. Check every data type, especially the two date fields
  9. Split Order ID on the dash and see what the parts contain
  10. Add a data source filter excluding 2014, then verify your totals drop
  11. Switch the connection to an Extract and note whether interaction feels faster
  12. Add an extract filter for 2016–2017 only and compare the extract's size
🏆 Challenge: build the fan-out yourself

Make a small two-column Targets table by hand — one row per region with a monthly target — and connect it to Orders. First as a join: sum the targets and watch the number come out roughly 2,500 times too large. Then as a relationship: watch it come out right. Finally, build a Sales vs Target view with a bullet chart. Having caused the bug deliberately once, you'll recognise it instantly when it happens for real.
