🧮 Calculated Fields

New data derived from existing fields. Ratios, categories, date maths, conditional logic — this is where Tableau stops being limited to the columns you were given and starts answering the questions you actually have.

📖 Terms on this page

Defined before they're used. Every term on the site is also in the glossary.

Calculated field
A new field you define with a formula. It joins the Data pane like any other.
Example: SUM([Profit])/SUM([Sales]) saved as Profit Margin.
📍 Analysis → Create Calculated Field
Row-level calculation
Runs once per row of the source data, before aggregation.
Example: [Profit]/[Sales] computes a margin for each of the 9,994 line items.
Aggregate calculation
Runs on already-totalled numbers. Nearly always what you want for ratios.
Example: SUM([Profit])/SUM([Sales]) gives West 14.9%; the row-level version gives 4.2%.
Null
A missing value — not the same as zero.
Example: A blank Sales cell means "unknown"; zero means "sold nothing".
ZN
A function that turns Null into 0.
Example: ZN([Profit]). Convenient, and often the wrong choice — decide deliberately.
Parameter
A single value the viewer can change, which calculations can read.
Example: A "Sales Threshold" slider that recolours a chart as you move it.

✏️ Creating a calculated field

Three routes to the same dialog: Analysis → Create Calculated Field, right-click in the Data pane, or the dropdown on any field → Create → Calculated Field.

Tableau validates as you type. Red means it won't save — and the message underneath usually tells you exactly what's wrong.

Syntax essentials

  • Field names go in [square brackets]
  • Text uses "double quotes"
  • Date parts use 'single quotes'
  • // starts a comment
  • Function names are case-insensitive; field names are not

Habits worth forming

  • Name fields for what they mean, not how they're built
  • Comment anything non-obvious — you will forget
  • Break long logic into several small fields
  • Use the right-hand panel to look up functions

⚠️ Row-level vs aggregate — the distinction that matters

Every calculation runs either on each row of the data or on the aggregated numbers in your view. Mixing them is the most common error in Tableau, and for ratios it produces genuinely wrong answers rather than errors.

Row-level, then averaged
AVG([Profit] / [Sales])

Computes a margin for every line item, then averages those percentages. A $5 order counts as much as a $5,000 one.

West region: 4.2%

→
Aggregate, then divided
SUM([Profit]) / SUM([Sales])

Totals both columns first, then divides. Every dollar weighted equally — the actual margin.

West region: 14.9%

Follow the arithmetic

Two rows of data, two computation paths, two different answers. Both look reasonable on screen.

The gap widens as your rows get more uneven. On real Superstore data the two answers differ by roughly 3×.

Neither errors — and they differ by a factor of three

For any ratio — margin, conversion rate, cost per unit — you almost always want SUM(a) / SUM(b), not AVG(a/b). The row-level version is an average of percentages, which weights a tiny order the same as an enormous one. It's the single most common way a Tableau dashboard ends up quietly reporting the wrong number.

// ✗ Won't even save — mixing the two
SUM([Profit]) / [Sales]
//   "Cannot mix aggregate and non-aggregate arguments"

// ✓ Both aggregated
SUM([Profit]) / SUM([Sales])

// ✓ Both row-level — Tableau aggregates the result later
[Profit] / [Sales]

A row-level field appears in the Data pane as a measure you can aggregate however you like. An aggregate calculation is already aggregated — you can't change it with the pill dropdown. That's a useful way to tell which you wrote.

➗ Basic calculations

// Ratios — aggregate both sides
SUM([Profit]) / SUM([Sales])           // profit margin
SUM([Sales]) / COUNTD([Order ID])       // average order value
SUM([Sales]) / SUM([Quantity])         // price per unit

// Row-level maths
[Sales] * (1 - [Discount])              // revenue after discount
[Sales] - [Profit]                       // cost of goods

// Aggregation functions
SUM()  AVG()  MIN()  MAX()  MEDIAN()
COUNT()      // number of rows
COUNTD()     // number of DISTINCT values
STDEV()  VAR()

// Rounding and absolutes
ROUND([Sales], 2)
ABS([Profit])
CEILING([Sales])   FLOOR([Sales])
COUNT vs COUNTD on Superstore

Each row is one line item, so COUNT([Order ID]) returns 9,994 — the number of lines, not orders. COUNTD([Order ID]) returns 5,009, the actual number of orders. Reporting the first as "orders" overstates it by roughly 2×, and it's an easy mistake to ship.

🔀 Conditional logic

IF — for conditions

// Basic
IF [Profit] > 0 THEN "Profitable"
ELSE "Loss"
END

// Several branches
IF [Sales] > 1000 THEN "High"
ELSEIF [Sales] > 500 THEN "Medium"
ELSE "Low"
END

// IIF — compact, two outcomes
IIF([Profit] > 0, "Profit", "Loss")

CASE — for exact matches

CASE [Region]
  WHEN "East"  THEN "Coastal"
  WHEN "West"  THEN "Coastal"
  ELSE "Inland"
END

// Assign a custom sort order
CASE [Ship Mode]
  WHEN "Same Day"     THEN 1
  WHEN "First Class"  THEN 2
  WHEN "Second Class" THEN 3
  ELSE 4
END
CASE when you can, IF when you must

CASE is faster and far more readable, but it only compares one field against exact values. Anything involving ranges, multiple fields, or AND/OR needs IF. The sort-order trick above is worth remembering — it's how you get Ship Mode to sort by speed rather than alphabetically.

Combining conditions

// AND / OR / NOT
IF [Profit] > 0 AND [Discount] = 0 THEN "Clean sale"
ELSEIF [Profit] < 0 AND [Discount] > 0.2 THEN "Over-discounted"
ELSE "Normal"
END

// IN — cleaner than a chain of ORs
IF [Region] IN ("East", "West") THEN "Coastal" ELSE "Inland" END

// A boolean field — returns True/False directly, no IF needed
[Profit] < 0

📅 Date calculations

// Difference between two dates
DATEDIFF('day', [Order Date], [Ship Date])     // days to ship
DATEDIFF('month', [Order Date], TODAY())     // months ago

// Extract parts
YEAR([Order Date])                  // 2017
MONTH([Order Date])                 // 11 — a number
DATENAME('month', [Order Date])    // "November" — text
DATEPART('quarter', [Order Date])  // 4

// Truncate to the start of a period — keeps it a real date
DATETRUNC('month', [Order Date])   // 1 Nov 2017
DATETRUNC('quarter', [Order Date])

// Shift a date
DATEADD('month', 3, [Order Date])
DATEADD('year', -1, [Order Date])     // same date last year

// Is it in the current year?
YEAR([Order Date]) = YEAR(TODAY())
DATETRUNC vs DATENAME

DATETRUNC returns a date — it sorts and charts correctly. DATENAME returns text, so "April" sorts before "January". Use DATETRUNC for anything on an axis, and DATENAME only for labels.

✂️ Text functions

// Extract
LEFT([Order ID], 2)                // "CA"
RIGHT([Order ID], 6)               // "152156"
MID([Order ID], 4, 4)              // "2016"

// Transform
UPPER([Category])   LOWER([Category])   PROPER([City])
TRIM([Customer Name])              // strip stray spaces
LEN([Product Name])                // character count

// Search and test
CONTAINS([Product Name], "Chair")  // True / False
STARTSWITH([Order ID], "CA")
FIND([Order ID], "-")               // position of first dash
REPLACE([Country], "United States", "USA")

// Combine — + concatenates in Tableau, not &
[City] + ", " + [State]

// Splitting on a delimiter (4th part, from the left)
SPLIT([Order ID], "-", 2)             // "2016"

You can often skip the calculation entirely: right-click a field in the Data pane → Split or Custom Split. Tableau writes the fields for you and they're easier to maintain.

🕳️ Handling nulls

// ZN — turn Null into zero
ZN([Profit])

// IFNULL — substitute anything
IFNULL([Region], "Unknown")

// ISNULL — test for it
IF ISNULL([Customer Name]) THEN "Missing" ELSE [Customer Name] END

// Guard against divide-by-zero — Tableau returns Null, not an error
IF SUM([Sales]) = 0 THEN NULL
ELSE SUM([Profit]) / SUM([Sales]) END
Null is not zero — decide deliberately

ZN() is convenient and often wrong. A missing sales figure means "we don't know"; zero means "they bought nothing". AVG skips nulls but counts zeros, so converting them silently drags your averages down. Only use ZN when zero is genuinely the right answer.

🎛️ Parameters — let the viewer decide

A parameter is a single value a user can change. On its own it does nothing; combined with a calculated field it becomes the most flexible control in Tableau.

  1. Create the parameter

    Right-click in the Data pane → Create Parameter. Name it Threshold, type Float, and set a range.

  2. Use it in a calculation

    IF [Sales] > [Threshold] THEN "Above" ELSE "Below" END

  3. Show the control

    Right-click the parameter → Show Parameter. A slider or input box appears beside the view.

  4. Drag the calculation onto Color

    Now moving the slider re-colours the whole chart live. That's a "what if" analysis in four steps.

// A measure swapper — one chart, four metrics
// Parameter "Select Measure": string list of Sales/Profit/Quantity/Discount
CASE [Select Measure]
  WHEN "Sales"    THEN SUM([Sales])
  WHEN "Profit"   THEN SUM([Profit])
  WHEN "Quantity" THEN SUM([Quantity])
  ELSE AVG([Discount])
END
The measure swapper is worth building once

Drop that calculation on Rows and give viewers the parameter control. One chart now serves four questions, and your dashboard needs a quarter of the space. It's the single highest-value parameter pattern in Tableau.

🧠 Key intuitions

  • Ratios use SUM(a)/SUM(b), never AVG(a/b)
  • You can't mix aggregate and row-level in one expression
  • COUNTD for orders, COUNT for line items
  • CASE for exact matches, IF for everything else
  • DATETRUNC keeps it a date; DATENAME makes it text
  • Null ≠ zero — ZN() is a decision, not a cleanup
  • + concatenates text in Tableau, not &
  • Name fields for meaning, and comment the non-obvious
  • Parameters + CASE = one chart, many questions

🏆 Challenge: an order quality score

Build a score from several criteria — 3 points for profit over $100, 2 for no discount, 1 for fast shipping — then bucket the total into Excellent / Good / Poor. Colour a scatter of Sales vs Profit by that score. The interesting part isn't the calculation; it's deciding what deserves points and defending it. Then check whether high-scoring orders cluster anywhere: a particular segment, region, or category. If they do, you've found something actionable.
