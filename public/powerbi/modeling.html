Data Modeling

What is Data Modeling?

Data modeling defines how tables relate to each other. Proper relationships enable accurate calculations across multiple tables.

Model View

Click Model View icon (left sidebar)

You'll see:
- Tables as boxes
- Lines connecting related tables
- Relationship indicators (1, *)

Drag fields between tables to create relationships

Relationship Types

One-to-Many (1:*)  - Most common
Example: One Category → Many Products

One-to-One (1:1)   - Rare
Example: Customer → CustomerDetails

Many-to-Many (*:*) - Use carefully
Example: Students ↔ Classes

Star Schema

Best practice for Power BI:

           ┌───────────┐
           │  Date     │
           └─────┬─────┘
                 │
┌───────────┐    │    ┌───────────┐
│ Product   │────┼────│ Customer  │
└───────────┘    │    └───────────┘
                 │
           ┌─────┴─────┐
           │  Sales    │  ← Fact table (center)
           │  (Facts)  │
           └───────────┘

Dimension tables = Descriptive (who, what, when)
Fact table = Transactional (numbers, events)

Creating Relationships

// Method 1: Drag and drop
In Model View, drag a field from one table
to the matching field in another table

// Method 2: Manage Relationships
Home → Manage relationships → New
- Select tables and columns
- Set cardinality (1:*, 1:1, *:*)
- Set cross-filter direction

// Auto-detect
Home → Manage relationships → Autodetect

Date Table

// Create a proper date table for time intelligence

Date Table = 
    CALENDAR(
        DATE(2014, 1, 1),
        DATE(2017, 12, 31)
    )

// Add useful columns
Year = YEAR('Date Table'[Date])
Month = FORMAT('Date Table'[Date], "MMMM")
Quarter = "Q" & QUARTER('Date Table'[Date])

// Mark as date table
Table tools → Mark as date table

Hiding Fields

// Hide technical columns from report view
Right-click column → Hide in report view

Good to hide:
- ID columns (used for relationships)
- Duplicate keys
- Technical fields users don't need

Hidden fields still work in DAX and relationships!

Key intuitions

  • Relationships flow from "one" side to "many" side
  • Filters propagate through relationships
  • Star schema = best performance and simplest design
  • Always create a dedicated Date table for time intelligence
  • Avoid circular relationships (A→B→C→A)
  • Hide technical columns to keep Fields pane clean

Practice

  1. Open Model View and explore the auto-detected relationships
  2. Create a Date table using CALENDAR function
  3. Add Year, Month, Quarter columns to the Date table
  4. Create a relationship between Date table and Order Date
  5. Hide the Row ID and Order ID columns from report view
