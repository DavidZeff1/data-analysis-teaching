Association Rule Mining

Itemsets, Support / Confidence / Lift, and the Apriori Algorithm — "People who bought X also bought Y"

What Is Association Rule Mining?

Association rule mining is an unsupervised technique that finds rules of the form "if $A$ then $B$" in transaction-style data. The classic example is market basket analysis: discovering that customers who buy diapers also tend to buy beer.

🛒 Typical Use Cases

  • Retail — product bundling, cross-sell, store layout
  • E-commerce — "frequently bought together"
  • Web usage — pages viewed in the same session
  • Bioinformatics — co-occurring gene mutations
  • Fraud / risk — claim items that tend to appear together

📦 The Data Shape

Input is a list of transactions, each a set of items: 

T1: {bread, milk, eggs}
T2: {bread, butter}
T3: {bread, milk, butter, beer}
T4: {milk, beer}
T5: {bread, milk, butter}

Itemsets & Rules

Itemset

Any set of items, e.g. $\{\text{bread}, \text{milk}\}$. A $k$-itemset contains $k$ items. A frequent itemset is one that occurs in at least min_support of transactions.

Association Rule

An implication $A \Rightarrow B$ where $A$ and $B$ are disjoint itemsets. $A$ is the antecedent (LHS); $B$ is the consequent (RHS). Example: $\{\text{bread}, \text{milk}\} \Rightarrow \{\text{butter}\}$.

The Core Metrics: Support, Confidence, Lift

Three numbers tell you how interesting a rule is. Each captures a different aspect.

1. Support — how common is the itemset?

Fraction of transactions that contain the itemset.

$\text{support}(A) = \dfrac{|\{T : A \subseteq T\}|}{|T|}$

In the example above, {bread, milk} appears in T1, T3 and T5 out of 5 transactions → $\text{support} = 3/5 = 0.6$.

Use to filter out rare combinations: rules with tiny support are usually noise.

2. Confidence — how reliable is the rule?

Conditional probability that $B$ appears in a transaction given $A$ does.

$\text{confidence}(A \Rightarrow B) = \dfrac{\text{support}(A \cup B)}{\text{support}(A)} = P(B \mid A)$

For $\{\text{bread}\} \Rightarrow \{\text{milk}\}$: bread appears in 4 transactions; bread and milk together in 3. Confidence $= 3/4 = 0.75$.

⚠️ Confidence alone is misleading: if the consequent is very popular on its own, any rule will look strong. That's why we need lift.

3. Lift — is the rule better than chance?

Ratio of observed support to what we'd expect if $A$ and $B$ were independent.

$\text{lift}(A \Rightarrow B) = \dfrac{\text{support}(A \cup B)}{\text{support}(A)\,\text{support}(B)} = \dfrac{\text{confidence}(A \Rightarrow B)}{\text{support}(B)}$

  • $\text{lift} = 1$ → $A$ and $B$ are independent (the rule is uninteresting)
  • $\text{lift} > 1$ → $A$ makes $B$ more likely (positive association)
  • $\text{lift} < 1$ → $A$ makes $B$ less likely (negative association)

Other useful metrics

  • Leverage = $\text{support}(A \cup B) - \text{support}(A)\text{support}(B)$ — additive form of lift; 0 means independence.
  • Conviction = $\dfrac{1 - \text{support}(B)}{1 - \text{confidence}(A \Rightarrow B)}$ — measures how much $A$ "implies" $B$; 1 means independence, $\infty$ means a perfect rule.

The Apriori Algorithm

Naively enumerating every possible itemset is exponential in the number of items. Apriori prunes the search space with a single, powerful observation:

The Apriori Property (downward closure):

If an itemset is frequent, then all of its subsets are also frequent. Equivalently: if an itemset is infrequent, no superset of it can be frequent — so we can stop exploring it.

The Algorithm

  1. Scan all transactions to find 1-itemsets with support ≥ min_support → $L_1$.
  2. From $L_k$, generate candidate $(k+1)$-itemsets by joining pairs that share $(k-1)$ items.
  3. Prune any candidate that has an infrequent subset (by the apriori property).
  4. Scan the data once more to count support for surviving candidates → $L_{k+1}$.
  5. Repeat until no new frequent itemsets are found.
  6. From all frequent itemsets, generate rules $A \Rightarrow B$ that meet min_confidence.

Python implementation (mlxtend)

import pandas as pd
from mlxtend.preprocessing import TransactionEncoder
from mlxtend.frequent_patterns import apriori, association_rules

transactions = [
    ['bread', 'milk', 'eggs'],
    ['bread', 'butter'],
    ['bread', 'milk', 'butter', 'beer'],
    ['milk', 'beer'],
    ['bread', 'milk', 'butter'],
]

# 1. One-hot encode into a boolean DataFrame (rows = transactions, cols = items)
te = TransactionEncoder()
df = pd.DataFrame(te.fit(transactions).transform(transactions), columns=te.columns_)

# 2. Find frequent itemsets
freq = apriori(df, min_support=0.4, use_colnames=True)

# 3. Derive rules and rank by lift
rules = association_rules(freq, metric="confidence", min_threshold=0.6)
rules = rules.sort_values("lift", ascending=False)
print(rules[["antecedents", "consequents",
             "support", "confidence", "lift"]])

✓ Strengths

  • Simple, interpretable rules
  • Pruning makes it tractable on real data
  • No need to specify the number of rules in advance

✗ Weaknesses

  • Multiple passes over the data — slow on very large datasets
  • Candidate generation can still explode with low min_support
  • For very large data prefer FP-Growth (one pass, tree-based)

Tuning the Thresholds

Parameter Too low Too high Sensible start
min_support Combinatorial explosion of rules; lots of noise You miss valid but niche patterns 0.01 – 0.05 for retail; raise until manageable
min_confidence Many spurious / weak rules Only obvious rules survive 0.5 – 0.7
min_lift Includes rules no better than chance Only very strong dependencies survive > 1 (filter); rank by lift afterwards

⚠️ Common Pitfalls

  • Trusting high-confidence, low-lift rules — they just reflect a popular item, not an association.
  • Setting min_support too low on a large catalogue — Apriori becomes intractable.
  • Reading rules as causation — they describe co-occurrence, not "X causes Y".
  • Forgetting to deduplicate transactions — repeated rows inflate support.
  • Mixing item granularities (specific SKUs and broad categories together) — produces meaningless rules.
