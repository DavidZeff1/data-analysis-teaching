Cosine Similarity

Measuring the Angle Between Vectors — The Workhorse of Recommenders, Search & NLP

What Is Cosine Similarity?

Cosine similarity measures how aligned in direction two vectors are, ignoring their magnitude. Two vectors pointing the same way score $1$; perpendicular vectors score $0$; opposite vectors score $-1$.

$\cos(\theta) = \dfrac{\mathbf{a} \cdot \mathbf{b}}{\|\mathbf{a}\|\,\|\mathbf{b}\|} = \dfrac{\sum_{i=1}^{n} a_i b_i}{\sqrt{\sum_{i=1}^{n} a_i^2}\,\sqrt{\sum_{i=1}^{n} b_i^2}}$

dot product divided by the product of L2 norms.

$\cos \theta = 1$

Same direction → identical orientation.

$\cos \theta = 0$

Orthogonal → no shared direction (no overlap).

$\cos \theta = -1$

Opposite directions. Rare for non-negative data (e.g. TF-IDF).

Why Angle Instead of Euclidean Distance?

Euclidean distance is affected by magnitude. Cosine similarity is not — which is exactly what you want when magnitude reflects size, not preference.

Example — comparing documents

Two TF-IDF vectors: a short tweet and a long article that cover exactly the same topics in the same proportions. Their magnitudes are very different (the article has more words), so Euclidean distance flags them as far apart. Cosine similarity correctly reports them as near-identical because their directions in word-space match.

✓ Use Cosine When…

  • You care about proportions not absolute values
  • Vectors are sparse / high-dimensional (text, user-item)
  • Magnitude reflects size, not meaning (long vs short documents)
  • You're working with embeddings (Word2Vec, sentence-transformers, OpenAI)

✗ Prefer Euclidean When…

  • Magnitude carries meaning (physical measurements, prices)
  • Features are low-dimensional and on the same scale
  • You're doing geometric clustering like K-Means on positions

Computing It in Python

From scratch with NumPy

import numpy as np

def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

a = np.array([1, 2, 3])
b = np.array([2, 4, 6])
print(cosine_similarity(a, b))   # 1.0 — same direction, different magnitude

Pairwise similarity matrix with scikit-learn

from sklearn.metrics.pairwise import cosine_similarity

# X is a (n_samples, n_features) matrix — dense or sparse
sim = cosine_similarity(X)   # shape (n_samples, n_samples)
# Compare two specific matrices: cosine_similarity(X, Y)

💡 Performance Tip

If you'll compute many similarities, L2-normalise once up front with sklearn.preprocessing.normalize(X). After that, cosine similarity is just the dot product $\mathbf{a} \cdot \mathbf{b}$ — much faster, and works seamlessly on sparse matrices.

Cosine distance vs cosine similarity

Many libraries use cosine distance = $1 - \cos(\theta)$ so smaller = more similar (matching the convention of other distance metrics). Watch for which one a function returns: sklearn.metrics.pairwise.cosine_similarity returns similarity; scipy.spatial.distance.cosine returns distance.

Worked Example — Comparing Documents

A classic NLP pipeline: turn text into TF-IDF vectors, then use cosine similarity to find the most similar documents to a query. 

from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.metrics.pairwise import cosine_similarity

docs = [
    "the cat sat on the mat",
    "a cat is sitting on a mat",
    "machine learning models are powerful",
    "deep learning is a subset of machine learning",
]
query = ["cats and mats"]

vec = TfidfVectorizer()
X      = vec.fit_transform(docs)
q_vec  = vec.transform(query)

sims = cosine_similarity(q_vec, X).flatten()
for i in sims.argsort()[::-1]:
    print(f"{sims[i]:.3f}  {docs[i]}")

The two cat-and-mat documents score high against the query; the machine-learning ones score near zero, even though they're longer.

Where Cosine Similarity Shows Up

🔍 Search & Information Retrieval

Rank documents by similarity to a query vector — the backbone of TF-IDF search and modern vector search engines.

🎬 Recommender Systems

Item-based and user-based collaborative filtering: similar users have similar rating vectors; similar items have similar user-rating patterns.

🧠 Embeddings & RAG

The standard metric for comparing sentence-transformer / OpenAI / Cohere embeddings — and the default in vector DBs (Pinecone, Weaviate, pgvector, FAISS).

📊 Clustering & Deduplication

Drop-in distance for clustering text/embedding data — pass metric="cosine" to AgglomerativeClustering or DBSCAN.

Mini Recommender — Item-Based Collaborative Filtering

Given a user-item rating matrix, find items similar to one the user already likes. 

import pandas as pd
from sklearn.metrics.pairwise import cosine_similarity

# rows = users, cols = items, values = ratings (0 = not rated)
ratings = pd.DataFrame({
    "Inception":    [5, 4, 0, 1],
    "Interstellar": [5, 5, 0, 2],
    "Toy Story":    [1, 2, 5, 4],
    "Finding Nemo": [2, 1, 5, 5],
}, index=["u1", "u2", "u3", "u4"])

# Item-item similarity: transpose so each row is an item's user-rating vector
item_sim = pd.DataFrame(
    cosine_similarity(ratings.T),
    index=ratings.columns, columns=ratings.columns,
)
print(item_sim["Inception"].sort_values(ascending=False))
# Interstellar will rank highest — same audience pattern.

Pitfalls & Variants

⚠️ Watch out for

  • Zero vectors are undefined ($\|0\| = 0$); filter them out or return 0.
  • Mean-centring matters for rating data: raw cosine treats unrated items as 0 (a strong dislike). Use adjusted cosine (subtract each user's mean rating) for collaborative filtering.
  • High dimensionality isn't a problem for cosine the way it is for Euclidean — that's one of its main attractions.
  • Negative values in embeddings can yield negative similarities; clip or shift if your downstream code assumes $[0, 1]$.
  • Cosine on TF (counts only) is dominated by stopwords — use TF-IDF or remove stopwords first.

Cousins worth knowing

  • Dot product — cosine without normalisation; faster, but magnitude leaks in.
  • Jaccard similarity = $|A \cap B| / |A \cup B|$ — for binary sets (tags, watched/not-watched).
  • Pearson correlation = cosine on mean-centred vectors. Use when each row has its own baseline (e.g. per-user rating scale).
  • Manhattan / Euclidean — magnitude-aware distances; better when scale is meaningful.
